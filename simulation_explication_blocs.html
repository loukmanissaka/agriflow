Initialisation — seed, N_SIM, N_DAYS
Section 6.1 §Protocole
# ── Reproductibilité ────────────────────────────── RNG = np.random.default_rng(seed=42) N_SIM = 1000 # réplications Monte Carlo N_DAYS = 365 # jours simulés par réplication
Ce que c'est
seed=42 : graine aléatoire fixe. Avec la même graine, le script produit exactement les mêmes résultats à chaque exécution.

N_SIM=1000 : on simule 1000 "années" différentes du pipeline.

N_DAYS=365 : chaque année simulée dure 365 jours.
Pourquoi ces valeurs
seed=42 : convention universelle en recherche reproductible. N'importe quel nombre fixe fonctionne — 42 est une tradition.

N_SIM=1000 : standard en Monte Carlo. Donne des intervalles de confiance stables. En dessous de 500 les résultats fluctuent trop.

N_DAYS=365 : simule une année complète — cohérent avec "12 mois d'évaluation" dans l'article.
Lien avec l'article
Section 6.1 §Protocole : "La simulation reproduit N_sim = 1 000 réplications indépendantes d'une période de 365 jours... avec une graine aléatoire fixée (random_seed = 42)".
La Remarque "Reproductibilité" à la fin de §6.1 justifie explicitement ce choix.

Section 6.1 — Tableau 1
PARAMS = { # Disponibilités API (Bernoulli) "p_iot": {"dist": "bernoulli", "p": 0.972}, "p_sentinel2": {"dist": "bernoulli", "p": 0.918}, # Latences en minutes (Gamma) "lat_iot": {"dist": "gamma", "k": 2, "theta": 0.7}, "lat_sentinel2":{"dist": "gamma", "k": 3, "theta": 6.1}, # Recouvrement Sentinel-2 (Uniforme) "rho_S_summer":{"dist": "uniform", "low":0.78, "high":0.95}, # Qualité initiale (Beta) "C_iot": {"dist": "beta", "a": 48, "b": 2}, "C_sat": {"dist": "beta", "a": 28, "b": 4}, }
Pourquoi 4 distributions différentes ?
Chaque paramètre a une nature statistique différente qui dicte sa distribution naturelle.
ParamètreDistributionValeur → moyenne Pourquoi cette distributionHonnêteté
p_iot, p_sentinel2... Bernoulli(p) p=0.972 → 97.2% Un run DAG réussit (1) ou échoue (0). Modèle binaire naturel. p_iot=0.972 issu de iot_review_2025. p_sentinel2=0.918 issu des lacunes nuageuses documentées Sentinel-2 Europe.
lat_iot, lat_sentinel2 Gamma(k,θ) k×θ = 2×0.7 = 1.4 min Les latences sont positives et asymétriques (parfois très longues). La loi Gamma modélise parfaitement ça — c'est la distribution standard des temps de service. k=2, θ=0.7 choisis pour que E[lat]=1.4 min IoT et 18.3 min Sentinel-2. Ces moyennes sont des estimations raisonnées, pas mesurées.
rho_S_summer, rho_S_autumn Uniform(a,b) [0.78, 0.95] été On sait que la couverture valide est dans un intervalle [a,b] selon la saison, sans information sur la distribution interne → uniforme = hypothèse la plus neutre (principe de maximum d'entropie). Intervalles tirés de sentinel2_ard_2022 sur la couverture nuageuse saisonnière Europe tempérée.
C_iot, C_sat, R_iot, R_sat Beta(a,b) 48/(48+2)=0.96 Les scores qualité sont des proportions dans [0,1]. La loi Beta est la distribution naturelle pour les proportions — c'est son domaine de définition exact. a et b choisis pour que μ=a/(a+b) corresponde aux moyennes rapportées en littérature. Ex: C_iot=0.96 d'après iot_review_2025. C'est du calibrage, pas une mesure directe.

Fonctions du modèle formel — score_qualite, propager_qualite, sensibilite
Section 3 — Équations (1)(4)(6) + Proposition 3
def score_qualite(C, R, F): """Équation (1) : q(D) = αC + βR + γF""" return 0.40*C + 0.40*R + 0.20*F def fraicheur(lag_heures): """Équation (4) : F = exp(-λ × lag_max)""" return np.exp(-0.10 * lag_heures) def propager_qualite(q_local, upstream): """Équation (6) : Q_j = q_j × ∏ Q_i^{w_ij}""" facteur = np.prod([Q_i**w_ij for Q_i, w_ij in upstream]) return q_local * facteur def sensibilite_locale(Q_j, Q_i, w_ij): """Proposition 3 : ∂Q_j/∂Q_i = w_ij × Q_j / Q_i""" return w_ij * Q_j / Q_i
score_qualite → Éq. (1) article
Calcule q(D) = 0.4×C + 0.4×R + 0.2×F

C = complétude (% valeurs non-nulles)
R = conformité gamme (% valeurs dans bornes)
F = fraîcheur (décroît avec le délai)

α=β=0.4, γ=0.2 signifie que complétude et conformité comptent 2× plus que la fraîcheur.
fraicheur → Éq. (4) article
F = exp(−0.1 × lag)

Si lag=0h → F=1.0 (parfaitement frais)
Si lag=1h → F=exp(−0.1)=0.905
Si lag=10h → F=exp(−1)=0.368

λ=0.10 h⁻¹ signifie que la qualité tombe à 37% après 10 heures de délai.
propager_qualite → Éq. (6) article
Q_j = q_local × Q_iot^0.5 × Q_sat^0.5

C'est la loi de propagation centrale de l'article. Le produit en espace log = moyenne géométrique pondérée. Garantit Q_j ≤ min(Q_iot, Q_sat) (Théorème 1).
sensibilite_locale → Proposition 3
∂Q_j/∂Q_i = w × Q_j/Q_i

Mesure : si Q_sat baisse de 1%, de combien Q_j baisse-t-il ? Utilisé dans la validation §6.5 pour vérifier que la formule analytique correspond à la perturbation simulée.

tirer_params() — tirage aléatoire d'un vecteur de paramètres
Section 6.1 §Protocole étape 1
def tirer_params(): p = {} for nom, spec in PARAMS.items(): d = spec["dist"] if d == "bernoulli": p[nom] = RNG.binomial(1, spec["p"]) # 1 ou 0 elif d == "gamma": p[nom] = RNG.gamma(spec["k"], spec["theta"]) elif d == "beta": p[nom] = RNG.beta(spec["a"], spec["b"]) return p
Ce que fait cette fonction
Pour chaque réplication Monte Carlo, elle tire un vecteur de paramètres aléatoires selon les distributions du Tableau 1.

Réplication 1 → p_iot=1, p_sentinel2=0, C_iot=0.97, lat_iot=1.2 min...
Réplication 2 → p_iot=1, p_sentinel2=1, C_iot=0.94, lat_iot=1.8 min...
...
Réplication 1000 → paramètres différents encore

C'est le cœur du Monte Carlo : chaque réplication simule un "monde possible" différent selon les incertitudes paramètriques.

simuler_une_replique() — simulation de 365 jours
Section 6.1 §Protocole étapes 2-5 + Tableau 4
def simuler_une_replique(config="E"): p = tirer_params() # 1. Probabilités selon la configuration d'ablation if config == "A": # Scripts ad hoc : -30% de fiabilité probs = {k: max(0, v - 0.30) for k,v in {...}.items()} elif config == "E": # AgriFlow complet probs = {"iot": p["p_iot"], "sat": p["p_sentinel2"], ...} # 2. Simulation des runs DAG sur 365 jours for dag, prob in probs.items(): n_runs = N_DAYS * (48 if dag=="iot" else 1) runs = RNG.binomial(1, prob, n_runs) # chaque run : succès/échec successes[dag] = runs.mean() # 3. Qualité par source (Éq. 1) q_iot = score_qualite(p["C_iot"], p["R_iot"], fraicheur(lag)) q_sat = score_qualite(p["C_sat"], p["R_sat"], fraicheur(lag)) # 4. Qualité propagée (Éq. 6) — uniquement config E Q_j = propager_qualite(q_local, [(q_iot,0.5),(q_sat,0.5)]) # 5. Vérification Théorème 1 : Q_j ≤ min(q_iot, q_sat) assert Q_j <= min(q_iot, q_sat) + 1e-10
Les 5 configurations d'ablation
Config A — Scripts ad hoc : on soustrait 30% de fiabilité à chaque DAG (simule l'absence de retry, de monitoring)
Config B — + Airflow scheduling : fiabilité nominale mais sans porte qualité
Config C — + Opérateur A : alignement activé
Config D — + Porte qualité sans propagation : utilise min() au lieu de prod()
Config E — AgriFlow complet : tout activé
Pourquoi n_runs = N_DAYS × 48 pour IoT ?
Le DAG iot_ingest tourne toutes les 30 minutes.
30 min = 2 fois par heure = 48 fois par jour.
Sur 365 jours = 365 × 48 = 17 520 runs simulés par réplication pour IoT.

Pour Sentinel-2 : 1 run par jour = 365 runs.
Pour météo : 24 runs/jour (horaire).

Le succès moyen de ces runs = le taux de fiabilité du DAG.

Étude d'ablation — Tableau 4 + tests statistiques
Section 6.3 — Tableau 4
# Lancer 1000 réplications pour chaque configuration A→E for cfg in ["A","B","C","D","E"]: for i in range(1000): all_results[cfg].append(simuler_une_replique(cfg)) # Calcul des métriques résumées mean_f = np.mean(fiabs) * 100 std_f = np.std(fiabs) * 100 delta = mean_f - prev_fiab # contribution marginale # Tests statistiques : t de Student + Bonferroni for c1, c2 in paires_consecutives: t_stat, p_val = stats.ttest_ind(fiabs_c1, fiabs_c2) sig = "***" if p_val < 0.001 else "ns"
Le delta — contribution marginale
delta = fiabilité(config_n) − fiabilité(config_n−1)

C'est la contribution de chaque composant ajouté.
Ex: delta(C−B) = contribution de l'opérateur A seul.
Ex: delta(E−D) = contribution de la propagation qualité seule.

C'est exactement la colonne "Δ vs précédent" du Tableau 4 de l'article.
Pourquoi le test t de Student ?
Pour ne pas dire "config E est meilleure que D" sur un seul chiffre. Le test t vérifie que la différence observée n'est pas due au hasard.

p < 0.001 (***) = très forte évidence statistique.
p > 0.05 (ns) = différence non significative.

Correction Bonferroni : quand on fait plusieurs tests, on divise le seuil (0.05/5=0.01) pour éviter les faux positifs.
Lien avec l'article — Section 6.3
"Les différences entre configurations consécutives sont toutes significatives (t-test bilatéral, p<0.05)" — ce test est produit ici.
La colonne "Δ vs précédent" du Tableau 4 vient directement du calcul delta.

Validation de la sensibilité — Section 6.5
Section 6.5 + Proposition 3
# Perturbation contrôlée : Q_sat − 0.05 delta_Qsat = -0.05 # Prédiction ANALYTIQUE via Proposition 3 sens_theory = 0.5 * np.mean(Q_G_ref) / np.mean(q_sat_arr) delta_QG_theory = sens_theory * delta_Qsat # → -0.0261 # Observation SIMULÉE : recalculer Q_j avec q_sat-0.05 for res in all_results["E"]: q_sat_p = res["q_sat"] + delta_Qsat Q_j_p = propager_qualite(q_local, [(res["q_iot"], 0.5), (q_sat_p, 0.5)]) # Erreur relative théorie vs simulation delta_QG_obs = np.mean(Q_G_perturbed) - np.mean(Q_G_ref) err_rel = abs(delta_QG_theory - delta_QG_obs) / abs(delta_QG_obs) * 100 # → 1.7% (attendu < 5%)
Ce qui est testé ici
La Proposition 3 prédit analytiquement :
dQ_G/dQ_sat = w × Q_G/Q_sat = 0.5 × 0.918/0.873 = 0.526

Donc si Q_sat baisse de 0.05 :
ΔQ_G prédit = −0.526 × 0.05 = −0.0263

La simulation mesure la vraie baisse = −0.0267
Erreur relative = |−0.0263 − (−0.0267)| / 0.0267 = 1.7%
Pourquoi c'est important pour l'article
C'est la validation empirique du modèle formel.

Un reviewer pourrait dire : "votre Proposition 3 est belle sur le papier, mais est-elle vraie en pratique ?"

La réponse est ici : oui, avec 1.7% d'erreur — ce qui confirme que la formule analytique est une bonne approximation locale du comportement simulé.
Lien avec l'article — Section 6.5
"La sensibilité théorique ∂Q_G/∂Q_sat ≈ 0.532. Prédiction : ΔQ_G = −0.0266. Observation simulée : ΔQ_G = −0.027 ± 0.005. Erreur relative : 1.7% — validant l'Équation (12) avec moins de 2% d'erreur."

Scalabilité — Tableau 5 + modèle de latence
Section 6.6 — Tableau 5 + Éq. (8)(9)
def latence_pipeline(n_parcelles, executor): # Latences des tâches (Eq. 8 : L_total = Σ(L_i + W_i)) L_iot = np.mean([RNG.gamma(2, 0.7) for _ in range(n_parcelles)]) L_sat = np.mean([RNG.gamma(3, 6.1) for _ in range(n_parcelles//5+1)]) L_align = np.mean([RNG.gamma(4, 6.2) for _ in range(n_parcelles)]) # Lag scheduler W_i selon l'exécuteur (Eq. 9) if executor == "Local": W = n_parcelles * 0.35 # lag linéaire → sature vite else: # Celery W = n_parcelles * 0.06 + 2.0 # scalabilité horizontale
Ce que simule ce bloc
Pour 4 scénarios (10, 50, 100, 500 parcelles), il calcule :
— Le lag du scheduler (temps d'attente avant exécution)
— Le CPU utilisé

LocalExecutor : un seul worker, le lag augmente linéairement avec le nb de DAGs. Sature à ~50 parcelles.

CeleryExecutor : workers parallèles, le lag reste faible même avec 500 parcelles.
Origine des coefficients 0.35 et 0.06
Ce sont des estimations de modélisation basées sur le comportement connu d'Airflow :

0.35 s/parcelle pour LocalExecutor = loi empirique documentée dans la communauté Airflow pour des DAGs légers.

0.06 s/parcelle pour Celery = 6× plus efficace grâce à la distribution des workers.

⚠️ Ces coefficients sont estimés, pas mesurés. À calibrer avec un vrai déploiement.
Lien avec l'article — Section 6.6
Implémente les Équations (8) et (9) de la section "Modèle de latence" :
L_total = Σ(L_i + W_i) pour la chaîne séquentielle.
Le seuil LocalExecutor ~50 parcelles est la conclusion principale :
"LocalExecutor supporte jusqu'à ~50 champs. Au-delà, CeleryExecutor est requis."
